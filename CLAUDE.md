# Familienkalender

Aggregierter Familienkalender als lokales Home-Assistant-Add-on. Zeigt die Kalender der Familie auf einem Touch-Display (Waveshare 15.6" HDMI, am Raspberry Pi 5) und in einer Web-UI in der HA-Seitenleiste (Ingress).

## Fachlicher Kern

- **Quellen:**
  - Marina (Google Calendar API): wird **komplett** angezeigt
  - Kunde (Google Calendar API, Workspace-Konto): **gefiltert**
  - Firma (Nextcloud CalDAV): **gefiltert**
  - Geburtstage (Google People API, Quellenart `google_contacts`): ganztägige Termine aus Kontakt-Geburtstagen (siehe Abschnitt „Quellenarten & Geburtstage")
- **Filterregel** für Kunde + Firma: Es erscheinen nur Termine, die für die Familie relevant sind — Termine, die in die Abendstunden hineinreichen oder dort beginnen (Grenze: 17:00 Uhr, im Admin konfigurierbar) sowie mehrtägige/übernachtende Termine. Reine Untertags-Meetings werden ausgeblendet — aber nur an Arbeitstagen: an Wochenenden und an gesetzlichen Feiertagen in Bayern (`app/holidays_bavaria.py`, Gauß-Osterformel, keine externe Dependency) werden alle Termine gezeigt; es zählt das lokale Startdatum (Europe/Berlin). Echte Termintitel werden angezeigt.
- **Aggregation intern** (kein Zurückschreiben in externe Kalender). Sync-Intervall ~5 Minuten.
- **Tages-Tags:** Pro Tag können Symbole (Smileys/Icons, kleine Auswahl) gesetzt werden — per Touch und Web. Nur lokale Speicherung.
- **Ansichten:** Monats- und Wochenansicht, blätterbar, touch-optimiert. Zusätzlich eine Strom-Ansicht (Balkonkraftwerk/Steckdosen via HA-API) und eine Wetter-Ansicht (Regenradar + Vorhersagegraph), zwischen denen und dem Kalender man am Display umschalten kann.
- **Adminbereich** (Web): Kalender-Zugänge, Google-OAuth-Verbindung, Abendgrenze, Icons, weitere Einstellungen.

## Architektur

- Python 3.12 / FastAPI-Backend, SQLite als lokaler Speicher
- Frontend: Vanilla JS/CSS, kein Framework, touch-optimiert (1920×1080)
- Verpackt als lokales HA-Add-on (`aarch64`, HA OS auf Raspberry Pi 5), Web-UI über **Ingress**
- Add-on-Root = Repo-Root (`config.yaml`, `Dockerfile` auf oberster Ebene), damit Deployment = Repo nach `/addons/familienkalender` auf den Pi synchronisieren

## Rechtemodell (Etappe 8)

- **Normale HA-Benutzer** (z. B. Marina, `familiendisplay` — darf Nicht-Admin sein): sehen das Panel (`panel_admin: false` in config.yaml) und nutzen Kalender, Tages-Symbole und Strom-Ansicht.
- **HA-Admins:** zusätzlich Verwaltung (`/admin`, `/api/admin/*`). Nicht-Admins bekommen dort 403 „Nur für Administratoren." (auf `/admin` als deutsche HTML-Seite); das Zahnrad im Header erscheint nur nach Bestätigung durch `GET /api/me`.
- **Vertrauenskette** (app/auth.py): Die IP-Allowlist lässt nur den Ingress-Proxy (172.30.32.2) und 127.0.0.1 zu → deshalb sind die vom Supervisor gesetzten Header `X-Remote-User-Id/-Name/-Display-Name` glaubwürdig (Quelle: supervisor/api/ingress.py `_init_header`). Einen Admin-Header gibt es nicht; die Admin-Gruppe wird per HA-WebSocket `config/auth/list` (`ws://supervisor/core/websocket`) aufgelöst, 60 s gecacht, Fehler = Nicht-Admin (fail closed). Anfragen von 127.0.0.1 **ohne** User-Header gelten als Admin (lokale Entwicklung, E2E, Healthcheck — echter Ingress setzt den Header immer) — **aber nur, wenn `SUPERVISOR_TOKEN` nicht gesetzt ist.** `SUPERVISOR_TOKEN` ist der zuverlässige Marker für „läuft im Add-on-Container"; ist er gesetzt, ist der Localhost-Fallback deaktiviert und header-lose 127.0.0.1-Anfragen gelten als Nicht-Admin (fail closed) — betrifft nur den Container-Healthcheck, echter Ingress-Traffic trägt immer den Header.
- **Bewusste Ausnahmen vom Admin-Gate** (Begründung in `docs/backlog.md`, Abschnitt „Bewusste Entscheidungen im Admin-Gate"): `POST /api/sync` bleibt für alle HA-Nutzer offen (schreibt nur in den internen Cache, gegen Mehrfachlauf per `SYNC_LOCK`/409 abgesichert, Ziele sind admin-konfiguriert); `/static/admin/*` bleibt ohne Gate lesbar (reine UI-Struktur, keine Secrets); der 60-s-Cache lässt einen Admin-Entzug entsprechend leicht verzögert wirken.

## Arbeitsregeln (verbindlich für alle Agenten und Sessions)

1. **TDD:** Erst Test schreiben, dann implementieren. Keine Etappe ist fertig ohne grüne Tests (`pytest`).
2. **Commits:** Jeder abgeschlossene Schritt wird **einzeln committet und gepusht** — direkt auf `main` (`git@github.com:roland1910/Familienkalender.git`).
3. **Quality-Gate:** `ruff check` (und Frontend-Lint, sobald vorhanden) muss vor jedem Commit sauber sein. Code gut lesbar und wartbar halten; nach jedem größeren Arbeitspaket läuft ein separater Quality-Review-Agent.
4. **Security:** Keine Secrets im Repo, in Commits oder im Chat. Credentials kommen zur Laufzeit aus der Admin-UI/HA-Optionen; für lokale Entwicklung aus `secrets.local.json` (gitignored). Security-Review bei jedem Meilenstein, der Credentials, Netzwerk oder Nutzereingaben berührt. **Frontend/Stored-XSS:** Event-Titel und -Location stammen aus fremden Kalendern/Einladungen — sie werden im Frontend ausschließlich via `textContent` gerendert, niemals via `innerHTML`.
5. **Sprache:** UI-Texte und Nutzer-Doku Deutsch; Code, Bezeichner und Kommentare Englisch.
6. **Deployment** auf den Pi macht ausschließlich der Orchestrator (Hauptsession), nicht Implementierungs-Agenten. Erprobte Prozedur (Pi seit 2026-07-21 über einen **USB-WLAN-Stick** unter **192.168.1.6**; der eingebaute WLAN-Chip ist abgeschaltet — vorher .4 bzw. LAN .3):
   ```
   git archive main | ssh -i ~/.ssh/id_ed25519 root@192.168.1.6 "tar -x -C /addons/familienkalender"
   ssh ... "ha store reload && ha addons rebuild local_familienkalender && ha addons restart local_familienkalender"
   ```
   Wichtig: `ha store reload` (nicht `ha addons reload`) macht Änderungen an config.yaml/neuen Add-ons sichtbar. Danach Logs prüfen: `ha addons logs local_familienkalender`. Das frühere Gotcha „Browser cacht altes JS nach Add-on-Update" ist seit Etappe 27 behoben: `/static/*` und die HTML-Einstiege (`/`, `/admin`) werden mit `Cache-Control: no-cache` ausgeliefert (`RevalidatingStaticFiles` + Header auf den HTML-Responses in `app/main.py`; Revalidierung per ETag → 304, bleibt schnell) — nach einem Deploy reicht ein normaler Reload. **Seit Etappe 30 gilt zusätzlich `Cache-Control: no-store` für ALLE `/api/*`-Antworten** (`ApiCacheControlMiddleware` in `app/main.py`, Pfadvergleich nach root_path-Strip, greift auch auf Fehler-Responses): der Kiosk-Browser hatte nach dem v0.26.0-Deploy ein heuristisch gecachtes altes `/api/config` (ohne `screensaver_default`) verwendet — Server-Defaults kamen nie an. Einzige Ausnahme: `/api/slideshow/image/*` behält bewusst `private, max-age=60`. Gürtel+Hosenträger: der `fetchConfig` in `api.js` sendet zusätzlich `{cache: "no-store"}`. Tests: `tests/test_cache_headers.py`.
7. **Rollen & Modelle:** Implementierung komplexer Logik → Fable/inherit; mechanische Arbeit & Quality-Review → Sonnet; Boilerplate/Doku → Haiku; Security-Review → Fable. Implementierungs-Agenten laufen im Hintergrund.

## Frontend

- Vanilla-ES-Module in `app/static/js/` (api, state, dates, events, colors, dom, popover, month-view, week-view, tag-picker, power-view, power-format, power-chart, weather-view, weather-chart, weather-format, weather-map, gestures, legend, view-memory, screensaver-memory, config-retry, slideshow-view, slideshow-format, slideshow-media, theme-memory, main), CSS in `app/static/css/`. **Kein Build-Schritt** — die Module laufen direkt im Browser.
- Quellen-Legende (`legend.js`): Farbpunkt + Name je aktivierter Quelle unter Monats-/Wochenansicht (nicht in der Strom-Ansicht), Farben identisch zu den Event-Chips (Auflösung siehe unten), Daten aus `GET /api/sources`.
- Strom-Ansicht: Umschalter oben rechts (`#mode-slot`), Daten von `GET /api/power` (`app/power.py`, HA Core API via `http://supervisor/core/api` + SUPERVISOR_TOKEN; lokal per `HA_API_URL`/`HA_API_TOKEN` übersteuerbar). Geräteliste als Setting `power_devices`, gepflegt im Admin unter Einstellungen. Layout (`power-view.js`): (1) **kompakte Kopfzeile** in EINER Zeile („Erzeugung X W · Verbrauch Y W · Bilanz Z W", Bilanz-Segment grün bei Überschuss / rot bei Netzbezug — dieselbe `balanceTile`-Logik wie zuvor, plus dezenter Gesamt-Zeitstempel), (2) **Verlaufsgrafik** „Erzeugung vs. Verbrauch", (3) Geräteliste (Watt rechts + kleiner Zeitstempel). Live-Werte pollen alle 15 s, die Grafik alle 60 s; beide Timer werden in `startPowerView`/`stopPowerView` verwaltet (kein Leak beim Verlassen).
- **Verlaufsgrafik (`/api/power/history`, Inline-SVG):** `GET /api/power/history?hours=<24|72|168>` (`app/power.py`) liefert `{hours, production:[{t,v}], consumption:[{t,v}]}` (t = Epoch-ms, v = Watt) aus der HA Core API `history/period/<start-iso>` mit `minimal_response&no_attributes`. Nur die **zwei fixen Aggregat-Entities** (`AGGREGATE_ENTITIES["production"]`/`["consumption"]`, Hoymiles-Wechselrichter + `sensor.stromverbrauch_gesamt`) — nie user-kontrolliert, kein SSRF/Injection. Härtung wie `/api/power`: 15 s-Timeout, Response-Größenlimit (`MAX_HISTORY_RESPONSE_BYTES` = 20 MB, Streaming-Zählung), 60 s-Cache **pro Fenster** (`_history_cache`, Lock gegen Thundering Herd), deutsche Fehlermeldung (502); `hours` wird auf die erlaubten Fenster geklemmt (sonst Default 24). **Downsampling PFLICHT** (7 Tage Rohdaten = tausende Punkte): `downsample()` reduziert je Serie auf max. `MAX_HISTORY_POINTS` (300) per gleichmäßigem Index-Sampling, ordnungserhaltend; nicht-numerische/`unavailable`-Punkte werden übersprungen. Frontend: Liniendiagramm mit zwei Serien (Erzeugung `--ok-border` grün, Verbrauch `--warn-text` amber), Y-Achse Watt mit Gridlines, X-Achse lokale Zeit, Legende (Farbpunkt+Name), Zeitraum-Buttons 1T/3T/1W (Default 1T, aktiver Button hervorgehoben, ≥44px). SVG rein via `document.createElementNS` (kein HTML-String); Chart-Geometrie (Skalierung, Achsen-Ticks, `linePath`, `niceCeiling`) als pure Funktionen in `power-chart.js` (node-testbar). Leere/fehlende Daten → dezenter Hinweis „Keine Verlaufsdaten" statt Absturz. Tests: `tests/test_power_history_api.py` (Parsing, Downsampling, hours-Validierung, Cache, HA-Fehler), `tests/js/power-chart.test.mjs` (Geometrie), E2E in `tests/e2e/test_power.py` (Kompaktzeile, SVG-Serien, Zeitraum-Wechsel, Leer-Hinweis).
  - **Zeitstempel „letzter Wert":** Jeder State liefert `last_updated` (Zeitpunkt des letzten State-Schreibens = Aktualität des Messwerts; bewusst nicht `last_changed`, das nur bei WERT-Änderungen vorrückt und bei konstanter Last veraltet wirkt). `/api/power` liefert `last_updated` je Gerät und je Aggregat-Kachel (ISO-String bzw. `None`, wenn Entity unavailable/unbekannt oder HA das Feld weglässt). Das Frontend zeigt die Uhrzeit klein/gedämpft (`--text-muted`) unter dem Wert — pure `formatLastUpdated(iso, now)` in `power-format.js` (heute → „HH:MM", gestern → „gestern HH:MM", älter → „TT.MM. HH:MM", leer/ungültig → „" = nichts gerendert; lokale Europe/Berlin-Zeit wie der Rest des Frontends). Auto-Refresh (15s) aktualisiert die Zeiten mit.
  - **Gerätenamen aus Home Assistant:** `/api/power` liefert je Gerät zusätzlich `friendly_name` aus `attributes.friendly_name` des States (`None`, wenn nicht vorhanden). Der im Admin (`power_devices`) gesetzte Name ist eine **optionale Übersteuerung**: Anzeigename-Priorität = konfigurierter Name → sonst `friendly_name` → sonst `entity_id` (pure `deviceDisplayName(configuredName, friendlyName, entityId)` in `app/static/admin/power-devices.js`, von `power-view.js` genutzt). `power_devices`-Zeilen dürfen daher nur die `entity_id` ohne „= Name" enthalten (Name dann leer → HA-Name); Parser/Backend/Storage lassen leere Namen zu, die `entity_id` wird weiter validiert. Aggregat-Kacheln haben feste deutsche Labels und tragen kein `friendly_name`.
- Adminbereich unter `/admin` (`app/static/admin/`: admin.html, main.js, api.js; Stile in `app/static/css/admin.css`), erreichbar über das Zahnrad im Kalender-Header. Gleiche Regeln: relative URLs, `textContent`-only, Deutsch. Backend-Endpunkte unter `/api/admin/*` (`app/admin.py`).
- Quellen-Farben: je Quelle optional konfigurierbar (Admin-Quellenzeile, `<input type="color">` + „Standardfarbe"), sonst Palette nach id. Auflösung zentral in `colors.js` (`colorForSource`/`colorForEvent`) — Chips, Wochenbalken, Popover und Legende nutzen dieselbe. **CSS-Injection-Regel:** Die Farbe wird in eine CSS-Variable interpoliert; server- (strikt `#rrggbb`, `is_valid_source_color`) UND clientseitig (Regex in colors.js, sonst Palette) validiert.
- Wochenansicht: leere Nachtstunden sind eingeklappt — das Raster beginnt um 08:00 bzw. zur vollen Stunde des frühesten Zeit-Termins der Woche (`gridStartHour` in week-view.js); kein Initial-Scroll mehr. Auto-Zoom: die Stundenhöhe ist die CSS-Variable `--hour-height` auf `.week-view` (alle vertikalen Positionen per `calc()`), von `applyWeekAutoZoom` so gesetzt, dass der sichtbare Bereich die Höhe füllt (Kiosk scrollt nicht); Minimum 24px, darunter Scroll-Fallback.
- Ansichts-Persistenz (`view-memory.js`): Ansicht (Monat/Woche), Anker-Datum und Modus (Kalender/Strom/Wetter) werden pro Gerät in localStorage gemerkt und beim Laden wiederhergestellt; der Heute-Button setzt auch den gespeicherten Anker. Gespeicherte Werte gelten als untrusted — ungültige fallen auf Defaults zurück. **Server-Default (Etappe 26):** Setting `default_view` (`month`|`week`, Default `month`, Admin-Select „Standard-Ansicht" unter Einstellungen, öffentlich lesbar via `GET /api/config`) bestimmt die Start-Ansicht für Geräte OHNE gespeicherten localStorage-Zustand — der Kiosk-Browser verliert seinen Speicher bei jedem Neustart. Priorität (`resolveInitialView` in view-memory.js, node-getestet): gültige pro-Gerät-Wahl gewinnt immer, sonst Server-Default, sonst Monat; main.js wartet den `/api/config`-Abruf VOR dem ersten Render ab (kein Flackern; angestoßen, wenn View-State ODER Schoner-Wahl fehlt), Fehler → Monat; der Server-Default wird bewusst nie in localStorage persistiert. **Boot-Rennen (Etappe 29):** Beim gleichzeitigen Start von Kiosk-Browser und Add-on (Pi-Reboot) schlägt der erste `/api/config`-Abruf fehl — dann rendern sofort die Fallbacks (Monat, Schoner aus) und `config-retry.js` (`createConfigRetry`, Zustandsautomat mit injizierbarem Fetch/Scheduler, node-getestet in `tests/js/config-retry.test.mjs`) versucht es im Hintergrund weiter (`RETRY_DELAYS_MS`: 5 Backoff-Versuche über ~60 s, danach stilles Aufgeben). Eine späte Antwort wird NUR angewandt, solange der Nutzer noch nicht interagiert hat (globales pointerdown/keydown/wheel setzt über `markInteraction` das Flag; auch ein laufender Fetch wird dadurch entwertet) — danach nie mehr Umschalten; wechselt die Ansicht spät, wird neu gerendert (`applyServerDefaults` in main.js).
- **Theme (Hell/Dunkel):** Alle UI-Chrome-Farben (Kalender **und** Admin) sind semantische CSS-Custom-Properties, einmalig auf `:root` in `calendar.css`/`admin.css` definiert (hell = Default): `--bg`, `--surface`, `--text`, `--text-muted`, `--border`, `--accent`, `--accent-contrast`, `--today-bg`, `--chip-fg`, `--chip-border`, `--warn-bg`/`--warn-text`, `--ok-*`/`--alert-*` (Strom-Kacheln), `--overlay`, `--shadow`, `--radius`. Keine fest kodierten Chrome-Farben mehr — nur die Diashow ist bewusst dauerhaft schwarz. Die Dark-Palette wird auf zwei Wegen aktiviert: `@media (prefers-color-scheme: dark)` (folgt dem System-/HA-„Auto"-Theme) **und** die manuelle Übersteuerung `:root[data-theme="dark"]` (gewinnt über die Media-Query); `data-theme="light"` erzwingt hell, `auto`/kein Attribut lässt die Media-Query entscheiden. **Quellen-Chip-Farben (`--source-color`) bleiben Theme-unabhängig** (eigener Hintergrund, heller Text auf hell wie dunkel). Manuelle Umschaltung pro Gerät via `theme-memory.js` (Muster wie view-memory: validiert `auto`|`light`|`dark`, ungültig → `auto`, node-getestet) — Toggle `#btn-theme` im Header-`#mode-slot` (zyklisch Auto/Hell/Dunkel, ≥44px). Das gespeicherte Theme wird per kleinem **inline** Head-Skript in `index.html`/`admin.html` **vor** dem ersten Paint auf `documentElement.dataset.theme` gesetzt (kein Hell-Aufblitzen; deferred Modul liefe zu spät). Default `auto`. Auch `forbidden.html` folgt per eigener Media-Query.

## Wetter-Ansicht (Etappe 35)

- **Zweck:** Dritter Modus neben Kalender und Strom (`#btn-mode-weather` im `#mode-slot`, Modus `weather` in `view-memory.js`; main.js schaltet die drei Modi über die gemeinsame `MODES`-Tabelle, damit immer nur die aktive Ansicht ihre Timer laufen lässt). Oben ein animiertes **Regenradar** auf München, darunter ein **Wettergraph** im Yr-Stil (Temperatur, Niederschlag, Wind), 24 h als Standard mit Umschaltung auf 48 h.
- **Datenquellen — alle drei frei und ohne API-Key, mit Auflagen, die im Code eingehalten und getestet sind:**
  - **MET Norway Locationforecast 2.0** (`https://api.met.no/weatherapi/locationforecast/2.0/compact`, fest auf München 48.1374/11.5755). Auflagen: (a) aussagekräftiger `User-Agent` mit Kontakt (`Familienkalender/1.0 github.com/roland1910/Familienkalender`) — ohne wird abgewiesen; (b) Antworten cachen und bedingt anfragen: Cache mindestens 30 min (`FORECAST_CACHE_TTL_SECONDS`), ein `Expires`-Header verlängert ihn (gedeckelt auf 6 h), ein Refetch schickt `If-Modified-Since` aus dem gemerkten `Last-Modified` und behandelt **304** als „Cache behalten, TTL verlängern"; **203** (veraltete Produktversion) gilt als Erfolg; (c) Quellenangabe in der UI.
  - **RainViewer** (Frame-Liste `https://api.rainviewer.com/public/weather-maps.json`, Kacheln unter `https://tilecache.rainviewer.com<path>/256/{z}/{x}/{y}/2/1_1.png`). Der `path` aus der Liste ist ein **Hash, nicht die Zeitmarke** — er wird nur aus der Liste übernommen, nie gebaut.
  - **OpenStreetMap** (`https://tile.openstreetmap.org/{z}/{x}/{y}.png`) als Grundkarte; Quellenangabe „© OpenStreetMap-Mitwirkende" ist Pflicht, und die Tile Usage Policy verlangt sparsame Abfragen → serverseitiger Kachel-Cache.
- **Backend** (`app/weather.py`, Muster wie `app/power.py`): Härtung durchgehend wie dort — Timeout, hartes Größenlimit auf dem **gestreamten** Body (eine fehlende/lügende `Content-Length` darf es nicht aushebeln), Lock gegen Thundering Herd, kurz gecachte Fehler (60 s) und deutsche Fehlermeldungen als **502**.
  - `GET /api/weather/forecast` → `{"points": [{t (Epoch-ms), temp_c, precip_mm, wind_ms, wind_dir_deg}, …]}` für ~48 h. **Einzelne fehlende oder nicht-numerische Felder werden `null`**, statt die ganze Stunde zu verwerfen (MET lässt Felder weg); nur ein unlesbares `time` überspringt den Eintrag. Es gibt bewusst **keinen `hours`-Parameter**: der Server liefert immer 48 h, das Frontend schneidet 24/48 rein clientseitig zu — ein Cache-Eintrag bedient beides.
  - `GET /api/weather/radar/frames` → `{"frames": [{id (Epoch-Sekunden), t (Epoch-ms)}, …]}`, die letzten `MAX_RADAR_FRAMES` (10) Bilder aus `radar.past`, 5 min gecacht. Kaputte Einträge werden einzeln übersprungen; eine leere Liste ist ein Fehler (502).
  - **Kachel-Proxy** `GET /api/weather/tile/base/{z}/{x}/{y}` und `GET /api/weather/tile/radar/{frame}/{z}/{x}/{y}`. Grund für den Proxy: das Frontend bleibt bei relativen URLs (Ingress-Regel), wir cachen höflich (Grundkarte 24 h, Radar 5 min, `MAX_CACHED_TILES` = 300 mit FIFO-Verdrängung) und umgehen CORS.
  - **SICHERHEITSREGEL zum Kachel-Proxy (getestet in `tests/test_weather_radar.py`):** (1) Die **Ziel-Hosts stehen fest im Code** — nichts aus der Anfrage wählt einen Host, es gibt also keine SSRF-Fläche. (2) `z`/`x`/`y` werden **strikt als reine Ziffernfolge** geparst (`_DIGITS`, max. 12 Stellen — `int()` allein würde `" 12"`, `"+12"` und sogar `"1_2"` schlucken) und danach gegen `ALLOWED_ZOOMS` und ein Kachelfenster um München (`MAX_TILE_RADIUS` = 4, `is_allowed_tile`) geprüft; alles andere ist **400 und löst nie eine externe Anfrage aus**. (3) Die **Frame-ID wird ausschließlich in der zuletzt von RainViewer gelieferten Liste nachgeschlagen** (`_radar_paths`, bei Bedarf on demand geholt); nur der dortige Pfad wird verwendet, eine unbekannte ID ist **404** ohne Upstream-Anfrage — ein Client kann nie einen Pfad einschleusen. (4) Weitergereicht wird nur, was upstream als `image/*` deklariert (ein durchgereichter HTML-Body wäre same-origin-Inhalt im Browser), mit Größenlimit (`MAX_TILE_BYTES`) und `X-Content-Type-Options: nosniff`.
  - **Ausnahme von der no-store-Regel:** `/api/weather/tile/` steht in `API_CACHE_EXEMPT_PREFIXES` (`app/main.py`, jetzt ein Tupel statt eines einzelnen Präfixes) und behält `private, max-age=…` — der Kiosk fordert dieselben Kacheln dauernd an.
- **Zoomstufen — die zentrale Fallgrube:** **RainViewer liefert Radarkacheln nur bis Zoomstufe 7**; darüber kommt ein Platzhalterbild „Zoom Level Not Supported" (immer dieselben 1370 Bytes, HTTP 200 — also kein Fehler, den man am Statuscode erkennt). Deshalb nutzt das **Radar die Stufen 5–7** (`RADAR_ZOOMS`, Default 7) und zeichnet seine Kacheln in **doppelter Größe** (`RADAR_TILE_PX` = 512), während die **Grundkarte eine Stufe tiefer** (6–8) in normaler Größe (256) geholt wird: identische Fläche, aber scharf statt hochskaliert. `ALLOWED_ZOOMS` im Backend spannt beides ab (5–8).
- **Kartenausschnitt pixelgenau statt Kachelraster (`weather-map.js`, node-getestet):** München liegt an einer beliebigen Stelle **innerhalb** seiner Kachel, ein Raster aus ganzen Kacheln würde die Stadt also bis zu eine halbe Kachel (~100 km) aus der Mitte schieben. `viewportTiles(zoom, tilePx, width, height)` berechnet daher wie eine echte Slippy-Map die Weltpixel-Position und legt die Kacheln mit Pixel-Offset ab; der Container (`.weather-radar-map`, feste Höhe `clamp(260px, 46vh, 520px)`) schneidet zu. Obergrenze `MAX_TILES_PER_LAYER` (40) gegen eine Anfrage-Lawine bei absurder Containergröße. Tests prüfen lückenlose Abdeckung, exakte Zentrierung auf München und dass **jede** angeforderte Kachel im Fenster liegt, das der Proxy zulässt (die Backend-Grenzen sind im JS-Test bewusst dupliziert, damit ein einseitiger Änderungsversuch auffällt).
- **Radar-Frontend** (`weather-view.js`): Grundkarte und je Zeitpunkt eine Radar-Ebene liegen deckungsgleich übereinander (`.weather-radar-base` / `.weather-radar-frame`), sichtbar ist immer genau eine (`.weather-radar-frame-active`, kurze Opacity-Blende). Animation als Endlosschleife (`FRAME_INTERVAL_MS` 500 ms, per `window.WEATHER_FRAME_MS` für Tests übersteuerbar; das neueste Bild wird `LAST_FRAME_HOLD_TICKS` länger gehalten), Play/Pause-Taste und Uhrzeit des gezeigten Frames. Die Kacheln werden **erst vorgeladen, dann läuft die Animation an** (`whenTilesSettled`, Muster wie die Diashow-Preloads) — mit `PRELOAD_TIMEOUT_MS`, damit eine kaputte Kachel die Animation nicht ewig blockiert. Zoom über −/+; eine `generation`-Zählnummer entwertet späte Preload-Rückläufe nach Zoomwechsel oder Ansichtswechsel. Da die Ebenen in Pixeln liegen, wird bei `resize` neu aufgebaut (entprellt; Listener und Timer werden in `stopWeatherView` abgeräumt).
- **Wettergraph** (`weather-chart.js` = reine Geometrie, node-getestet; SVG-Bau in `weather-view.js` per `createElementNS`, nie HTML-Strings): Temperaturkurve (links °C), Niederschlag als Balken (rechts mm), Wind als eigene Zeile unter der Zeitachse aus Richtungspfeil und Zahl (aus 2 m Entfernung besser lesbar als eine dritte Kurve). Details: die **Temperaturkurve bricht an Stunden ohne Messwert ab** (neues `M`) statt eine gerade Linie über die Lücke zu ziehen; die **Temperaturachse rastet auf runde Schritte ein** (`TEMP_INTERVALS` = 4, Schrittweite aus `[1,2,5,10,20]` — sonst stünden dort Werte wie 10/14/18/21/25); die Niederschlagsachse ist eine „nice" Obergrenze ab 1 mm und wird mit **zwei** Nachkommastellen beschriftet, damit die Viertelschritte einer 1-mm-Achse stimmen (0,25 statt 0,3); der **Windpfeil zeigt, wohin** der Wind weht (`windArrowRotation` = MET-`wind_from_direction` + 180°, Konvention wie bei Yr). Zeitraum-Buttons 24 h (Standard) / 48 h, ≥44 px, aktiver hervorgehoben — wie die 1T/3T/1W-Buttons der Stromgrafik. Achsenbeschriftung im 48-h-Fenster mit Wochentag-Präfix (`weather-format.js`).
- **Auto-Aktualisierung:** Vorhersage alle 30 min, Radar-Frames alle 5 min; beide Timer plus der Animations-Timer gehören `weather-view.js` und werden in `stopWeatherView` abgeräumt (kein Leak beim Verlassen — siehe power-view).
- **Ausfall einer Quelle nimmt nie die ganze Ansicht mit:** Fällt die Vorhersage aus, zeigt der Graph die deutsche Meldung und das Radar läuft weiter; fällt das Radar aus, steht der Hinweis über der Karte und der Graph bleibt. **CSS-Falle (realer Befund beim Sichtprüfen):** `.weather-radar-hint` braucht ein explizites `[hidden] { display: none }` — das `display: flex` der Klasse gewinnt sonst gegen das `hidden`-Attribut aus dem UA-Stylesheet, und der Ladehinweis bleibt für immer über der fertigen Karte liegen.

## Foto-Diashow (Kiosk-Bildschirmschoner)

- **Zweck:** Am Touch-Display läuft nach Leerlauf eine Vollbild-Diashow (Fotos und — optional, siehe Etappe 33 — Videos) als Bildschirmschoner. Die Dateien liegen auf der CIFS-Netzwerkfreigabe „Photos", die Home Assistant unter `/media` einbindet; das Add-on liest sie **schreibgeschützt** (config.yaml `map: media:ro`, apparmor.txt: `/media` nur `r`, `deny /media/** wklx`).
- **Backend** (`app/slideshow.py`, Muster wie `tags.py`): Setting `slideshow_dirs` (JSON-Liste von Ordnern unterhalb `/media`). Pfad-Validierung gegen Traversal per `os.path.realpath` + `commonpath` (`normalize_media_dir` — fängt `../` und Symlinks ab, die aus `/media` herauszeigen); auf Lese- **und** Schreibpfad validiert. Medien-Index: rekursiver `os.scandir`-Scan (`iter_media`; jpg/jpeg/png/webp und — immer, siehe Etappe 33 — mp4/mov/m4v, case-insensitive; Symlinks **nicht** gefolgt; `#recycle`/`@eaDir`/versteckte Ordner übersprungen; Fehler je Ordner isoliert; Obergrenze `MAX_INDEXED_PHOTOS = 100000`). Der Scan läuft in `asyncio.to_thread` (blockiert den Event-Loop nicht — die reale Freigabe hat ~114k Dateien) und ist per Lock serialisiert; ausgelöst beim Speichern und stündlich (`periodic_photo_scan`, nur mit `SLIDESHOW_SCAN=1` in der Lifespan aktiv, damit Tests/Dev ohne `/media` nicht scannen). Tabelle `photos (id, path UNIQUE, mtime, shown, kind)`.
- **Schutzregel gegen Index-Verlust (Etappe 31):** Nach einer Systemwiederherstellung startete das Add-on, bevor HA die CIFS-Freigabe unter `/media` eingebunden hatte — der Scan fand 0 Fotos, ersetzte den ~114k-Index durch die leere Menge, und der Kiosk zeigte nur noch Schwarz (`/api/slideshow/next` → 404). Deshalb prüft `_scan_sync` jetzt je konfiguriertem Ordner die **Verfügbarkeit** (`_dir_is_available`: existiert als Verzeichnis UND `os.scandir` öffnet fehlerfrei — ein vorhandener, aber unlesbarer Ordner zählt als nicht verfügbar). Semantik: (a) **kein** Ordner konfiguriert → Index wird geleert (legitimer Wunsch); (b) **alle** konfigurierten Ordner nicht verfügbar → Lauf wird ÜBERSPRUNGEN, Index bleibt unverändert, Warnung im Log; (c) **teilweise** verfügbar → nur die Fotos unterhalb der verfügbaren Ordner werden neu eingelesen, Einträge unterhalb nicht verfügbarer konfigurierter Ordner werden unverändert übernommen (kurzer Netzwerk-Aussetzer darf den Index nicht schrumpfen); Einträge, die unter KEINEM konfigurierten Ordner mehr liegen, fallen wie bisher weg; (d) erreichbarer, wirklich leerer Ordner → liefert weiter 0 Fotos (Verfügbarkeit, nicht Leere, ist das Kriterium). Der Ausgang wird als Setting `slideshow_scan_status` (`{unavailable_dirs, skipped}`) persistiert und von `GET /api/admin/slideshow` als `unavailable_dirs`/`scan_skipped` mitgeliefert; die Admin-Sektion zeigt daraus eine dezente deutsche Warnung (`scanWarningText` in `slideshow-path.js`, node-getestet). Tests: `tests/test_slideshow.py::TestScanAvailabilityGuard`.
- **Rotation mit Gedächtnis:** `GET /api/slideshow/next` liefert ein zufälliges Medium mit `shown=0`, markiert es `shown=1` (`{id, name, kind, taken, folders}`); sind alle gezeigt, wird `shown` zurückgesetzt und neu gezogen — ein voller Durchlauf vor Wiederholung, überlebt Neustart (Zustand in der DB). `MEDIA_ROOT` env übersteuert den Wurzelpfad für Tests.
- **Bild-/Video-Endpoint:** `GET /api/slideshow/image/{id}` streamt via `FileResponse` (Content-Type nach Endung, `Cache-Control: private, max-age=60`). Zugriff **nur über DB-id**, nie Pfad-Parameter; unbekannte id oder verschwundene Datei → 404 (verschwundene Datei wird aus dem Index entfernt). Läuft hinter Ingress + IP-Allowlist. **HTTP-Range/Teilanfragen** beherrscht Starlettes `FileResponse` (gepinnt 1.3.1) bereits von sich aus: `Accept-Ranges: bytes`, 206 + `Content-Range` bei `Range`-Header, 416 bei unerfüllbarem Bereich, 200 ohne Header. Kein Eigenbau — aber durch `tests/test_slideshow.py::TestRangeRequests` festgenagelt, damit ein Dependency-Update das nicht still verliert (ohne Range würde der Browser Videos nicht streamen/spulen können).
- **Frontend** (`slideshow-view.js`): Vollbild-Overlay, `object-fit: contain` auf Schwarz, zwei gestapelte Layer für sanfte Überblendung (opacity-Transition, 800ms), nächstes Bild vorab geladen (kein schwarzer Blitz), Wechsel alle `SLIDE_INTERVAL_MS` (30s; per `window.SLIDESHOW_INTERVAL_MS` für Tests übersteuerbar). Dateiname nur via `textContent`.
  - **Die Foto-Layer MÜSSEN echte `<img>`-Elemente sein — niemals `background-image`-Divs (Etappe 32, realer Befund am Kiosk):** CSS-Hintergrundbilder ignorieren das EXIF-Orientation-Tag, dadurch wurden Hochformat-Fotos um 90° gedreht angezeigt. `<img>` wertet es aus; `image-orientation: from-image` steht zusätzlich explizit im CSS (Absicht dokumentieren + Schutz gegen abweichende Engine-Defaults — der Kiosk nutzt WebKitGTK/luakit). Der noch unsichtbare Layer dient selbst als Preload (`loadIntoLayer` setzt `src` und wartet auf `load`, erst danach die Überblendung); ein bereits gesetztes, fertig geladenes `src` wird erkannt (identische URL feuert `load` nicht zuverlässig erneut — kommt bei kurzer Rotation vor). Badges/Caption liegen per `z-index: 1` über den Layern (`z-index: 0`). Regressionstest: `tests/e2e/test_slideshow_e2e.py::test_photo_layers_are_images_honouring_exif_orientation` (Tag-Name `IMG`, computed `image-orientation`/`object-fit`, kein `background-image`, Badge-Stacking).
- **Videos in der Diashow (Etappe 33):** Rolands Handy-Videos liegen mit in den Foto-Ordnern und laufen in der Diashow mit. Formate `.mp4`, `.mov`, `.m4v` (`VIDEO_EXTENSIONS`), Content-Types `video/mp4` bzw. `video/quicktime`; **keine Transkodierung, keine neue Dependency** — die Datei wird unverändert ausgeliefert, der Kiosk-Browser spielt sie oder meldet Fehler.
  - **Videos werden IMMER mit indiziert, unabhängig vom Schalter.** Sonst müsste jedes Umschalten die ganze Freigabe (~114k Dateien) neu einlesen. Der Schalter steuert ausschließlich die AUSLIEFERUNG.
  - **Setting `slideshow_videos`** (`on`|`off`, Default **`off`** — konservativ, weil Video-Dekodierung über die CIFS-Freigabe auf dem Pi ruckeln kann; Muster wie `screensaver_default` in `app/settings.py`, Re-Validierung beim Lesen). Bei `off` liefert `/api/slideshow/next` nur Bilder.
  - **Art im Index:** Spalte `photos.kind` (`image`|`video`), additive + idempotente Migration (`ALTER TABLE … DEFAULT 'image'` — Bestandszeilen stammen aus reinen Bild-Scans, sind also korrekt befüllt). Bewusst eine eigene Spalte statt Ableitung aus der Endung: die Rotation muss **in SQL** nach Art filtern; eine LIKE-Kette über die Endungs-Whitelist würde diese Liste in SQL duplizieren und jeden Index aushebeln. Die Art wird genau einmal entschieden, von derselben Whitelist, die die Datei beim Scan zugelassen hat (`media_kind` in `app/slideshow.py`).
  - **Rotation korrekt bei ausgeschalteten Videos:** `pick_next_photo(kinds=…)` filtert Auswahl **und** „alles gezeigt"-Reset auf die zulässigen Arten. Die nicht zulässigen Zeilen sind damit unsichtbar — „alles gezeigt" entscheidet sich allein über die Bilder, die Rotation bleibt nie an einem nie wählbaren Video hängen. Weil der Reset nur die zulässigen Arten zurücksetzt, wiederholt ein späteres Einschalten auch nicht die gerade gezeigten Bilder.
  - **Frontend** (`slideshow-view.js`): Bei `kind == "video"` rendert die Ebene ein `<video>` statt `<img>` — `muted`, `playsinline`, `autoplay`, **ohne** Bedienelemente, gleiches `object-fit: contain`, gleiche Überblendung, gleiche Badges darüber. Die Ebene wird nur getauscht, wenn sich die Art ändert (reiner Fotobetrieb = keine DOM-Änderung). Umgeblendet wird erst nach `loadeddata` (erstes Bild da, kein schwarzer Blitz). **Ablauf/Timing:** Video läuft bis `ended` → sofort nächstes Medium; Obergrenze `MAX_VIDEO_MS` (60s, per `window.SLIDESHOW_MAX_VIDEO_MS` testbar) als einziger Schutz gegen einen Clip, der weder endet noch Fehler meldet (hängender CIFS-Read); Lade-/Wiedergabefehler (nicht unterstützter Codec) → nach `VIDEO_ERROR_RETRY_MS` (1s, kurz aber bewusst nicht 0 — sonst Hot-Loop bei lauter kaputten Dateien) das nächste Medium. Bilder unverändert bei `SLIDE_INTERVAL_MS`. **Die Diashow bleibt NIE stehen:** jeder Pfad (Fetch-Fehler, Ladefehler, Abspielfehler, Überlänge) endet in einem geplanten Weiterschalten — deshalb ist aus dem `setInterval` ein pro Medium neu gesetztes `setTimeout` geworden. Genau einer der drei Video-Ausgänge darf schalten (`settled`-Flag); Listener werden beim Ebenenwechsel und in `stopSlideshow` abgeräumt und das Video pausiert (kein Leak, kein Weiterdekodieren). Reine Regeln (Art, Dauer, Fehler-Wartezeit) in `slideshow-media.js` (DOM-frei, node-getestet).
  - **Admin:** Select „Videos abspielen" (An/Aus) mit eigenem Speichern-Button in der Sektion „Diashow (Kiosk)" plus Hinweistext (immer mitgelesen; kann auf schwacher Hardware ruckeln; stumm, ohne Bedienelemente, max. 1 Minute). `GET/PUT /api/admin/slideshow` kennen `videos` — ungültiger Wert → 400, **weggelassen = unverändert**, damit die Ordner-Buttons die Einstellung nicht überschreiben. Ein Umschalten löst bewusst KEINEN Rescan aus.
- **Aufnahmedatum & Ordnerpfad (Etappe 28):** `GET /api/slideshow/next` liefert zusätzlich `taken` (strukturiert-partielles Aufnahmedatum `{year, month?, day?, hour?, minute?}` oder `null`) und `folders` (Ordnersegmente unterhalb der Media-Wurzel, ohne Dateinamen). `photo_taken_at` (app/slideshow.py) ermittelt das Datum zur Serve-Zeit mit Prioritätskette: (1) EXIF DateTimeOriginal/DateTime — eigener minimaler, bounds-gecheckter JPEG/TIFF-Parser (pure Python, keine neue Dependency, max. 256 KB gelesen, jede Ausnahme → None; nur JPEG, PNG/WebP tragen kein klassisches EXIF), (2) Dateinamens-Muster (`YYYYMMDD_HHMMSS`, `YYYY-MM-DD HH.MM.SS`, `IMG-YYYYMMDD-WA…`, nacktes `YYYYMMDD`; Jahr 1990..heute+1 validiert, valide Datum-Teile bleiben bei kaputter Zeit erhalten), (3) exakter Jahres-Ordner `(19|20)\d\d` im Pfad. Bei **Videos** greifen nur (2) und (3) — EXIF ist JPEG-only, und genau das tragen Handy-Videonamen (`VID_20190816_173005.mp4`). Frontend: zwei dezente Badges im Letterbox-Bereich (`.slideshow-taken` „16.08.2019 17:30"/„16.08.2019"/„2019", `.slideshow-folders` „Photos › 2019 › Urlaub"; Anordnung seit Etappe 34 senkrecht, siehe unten), Formatierung pure in `slideshow-format.js` (node-getestet), Wechsel synchron mit der Überblendung, Fremd-Strings nur via `textContent` — ist nichts auslesbar, wird das Badge komplett ausgeblendet (Rolands Vorgabe: dann steht dort einfach nichts).
- **Senkrechte Einblendungen am Bildrand (Etappe 34):** Die drei Text-Overlays stehen um 90° gedreht an den Bildschirmkanten, damit sie möglichst wenig vom Foto verdecken. **Linker Rand:** `.slideshow-folders` (Ordnerpfad), Leserichtung **von unten nach oben**, oben am Bildrand verankert — der Text ENDET oben, längere Pfade wachsen nach unten. **Rechter Rand oben:** `.slideshow-taken` (Aufnahmedatum), Leserichtung von oben nach unten, beginnt in der oberen Ecke. **Rechter Rand unten:** `.slideshow-caption` (Dateiname), gleiche Leserichtung, endet in der unteren Ecke (senkrechtes Äquivalent des früheren rechtsbündigen Untertitels; die Leserichtung der benachbarten Datums-Einblendung gab den Ausschlag, Rolands „rechtsbündig" ist am gedrehten Text mehrdeutig — im CSS kommentiert). Technik: **`writing-mode: vertical-rl` statt rohem `transform: rotate()`** — nur so folgen Zeilenumbruch, `text-overflow: ellipsis` und die Box-Größe dem gedrehten Text. Für „von unten nach oben" kommt zusätzlich `transform: rotate(180deg)` dazu (`sideways-lr` beherrscht WebKit nicht, und der Kiosk läuft auf WebKitGTK/luakit); eine 180°-Drehung um die eigene Mitte bildet die Box auf sich selbst ab, die Verankerung bleibt also erhalten. Weil der Text jetzt senkrecht läuft, begrenzt `max-height` die Länge (Ellipsis am Textende) und das Padding ist getauscht (`14px 6px`); die beiden rechten Overlays sind auf je knapp die halbe Höhe begrenzt und können sich damit nie überlappen. Farben, `z-index: 1` über den Ebenen und `pointer-events: none` unverändert. Test: `tests/e2e/test_slideshow_e2e.py::test_overlay_texts_run_vertically_along_the_screen_edges` (computed `writing-mode`/`transform`, Bounding-Boxen an den erwarteten Kanten, kein Überlappen rechts).
- **Bildschirmschoner pro Gerät** (`screensaver-memory.js`, localStorage wie view-memory): Toggle (`#btn-screensaver` im `#mode-slot`, Symbol seit Etappe 30 Bilderrahmen 🖼️ statt Kamera — die Kamera sah dem Kalender-Symbol 📅 des Mode-Umschalters am Kiosk zu ähnlich; die freistehenden Icon-Buttons Theme/Schoner rendern in 22px), Default **AUS**. Wenn AN: nach `IDLE_TIMEOUT_MS` (180s, per `window.SCREENSAVER_IDLE_MS` für Tests übersteuerbar) ohne pointerdown/keydown/wheel startet die Diashow; jede Interaktion beendet sie → Kalender im vorigen Zustand. Idle-Watcher pollt sekündlich (kein Timer-Reset pro Event). **Server-Default (Etappe 29):** Setting `screensaver_default` (`on`|`off`, Default `off`, Admin-Select „Diashow-Bildschirmschoner standardmäßig" unter Einstellungen, öffentlich lesbar via `GET /api/config` neben `default_view`) bestimmt den Zustand für Geräte OHNE eigene Wahl — der Kiosk verliert seinen localStorage bei jedem Neustart, soll den Schoner aber automatisch AN haben. Kommt `/api/config` erst spät (Boot-Rennen, siehe Ansichts-Persistenz), wird der Schoner nachträglich scharf geschaltet — aber nur ohne vorherige Nutzer-Interaktion. - **Eigener TLS-Listener, extern erreichbar:** Der Feed läuft als separater, minimaler ASGI-Baum (`app/feed_app.py`) auf Container-Port 8100 → Host-Port 8098 (`ports` in config.yaml); der Router darf extern 8098 dorthin weiterleiten (`https://rnd.ignorelist.com:8098/…`). `app/serve.py` (`python -m app.serve`, gestartet von run.sh) fährt Haupt-App (8099, Ingress) und Feed-Listener in EINEM Prozess/asyncio-Loop; TLS mit den HA-Zertifikaten aus `/ssl` (Options `ssl_certfile`/`ssl_keyfile`, Default fullchain/privkey; fehlen sie, startet nur der Feed nicht, stündlicher mtime-Check erkennt Erneuerung und startet den Listener neu). Die Haupt-App ist strikt ingress-only (keine Feed-Ausnahme mehr in der IP-Allowlist).
- **Härtung im Feed-Listener** (`app/feed_app.py`, Tests in `tests/test_feed_app.py`): URL-Token (settings-Key `feed_token`, `secrets.token_urlsafe(32)`, Vergleich per `compare_digest`) als einzige Auth, falsches/fehlendes Token → 404; nur GET/HEAD (405 sonst); Security-Header auf jeder Antwort; Rate-Limit 30/min pro IP + 300/min global; 15-Minuten-Lockout nach 10 Token-Fehlversuchen (absolut, auch bei gültigem Token); kein Access-Log (Token in der URL). Abo-URL (https, Setting `feed_public_host`, Default Request-Host) + Rotation in der Admin-Sektion „Kalender-Abo" (`/api/admin/feed`).
- **Dubletten-Zusammenführung (Etappe 18, nur im Feed):** Derselbe Termin kann in mehreren Feed-Quellen liegen (z. B. ein Meeting in Roland@Xalt (Google) UND in MoreValue (Nextcloud)). Im Abo erscheint er nur EINMAL. Kriterium: gleicher **Titel + Start + Ende + all_day-Flag**. Titel normalisiert (Groß/Klein egal, äußere Whitespaces getrimmt, Mehrfach-Whitespace zusammengefasst — `normalize_title`); Vergleich auf den ROHEN Titeln (der `shortcode`-Präfix wird erst danach vorangestellt). Start/Ende: Zeit-Termine als UTC-Instant (gleicher Moment in verschiedenen Zonen = Dublette; tz-lose datetime wird als Europe/Berlin gelesen, nicht als Prozess-Zeitzone), all-day per Datum; all_day vs. timed verschmilzt nie. **Location bewusst NICHT im Schlüssel** (Rolands Vorgabe — dieselbe Termin-Location weicht zwischen Kalendern oft leicht ab). **Dedup ist strikt quellenübergreifend:** Zwei Events werden nur zusammengeführt, wenn sie aus VERSCHIEDENEN Quellen (`source_id`) stammen. Zwei gleiche-Key-Events DERSELBEN Quelle (unterschiedliche `uid`) bleiben IMMER beide erhalten — sie sind echte, verschiedene Termine (z. B. zwei Kontakte „Oma"/„Müller" mit Geburtstag am selben Tag aus einer Kontakte-Quelle). Gewinner pro Gruppe ist die QUELLE mit höherer `feed_priority` (Integer-Spalte an `sources`, Default 0, Bereich −100..100, im Admin je Quelle als Zahlenfeld „Vorrang (Abo-Dubletten)"), bei Gleichstand deterministisch die niedrigere `source_id`; ALLE Events der Gewinnerquelle in der Gruppe bleiben, alle anderen Quellen fallen für diesen Key weg. So kann Vorrang nie mehrere echt verschiedene Events einer Quelle durch EIN Event einer anderen Quelle ersetzen. Die stabile UID des Gewinners bleibt erhalten. Reine Helfer `normalize_title`/`dedupe_feed_events` in `app/feed.py`, in `build_feed` VOR der VEVENT-Erzeugung angewandt. **Nur der Feed** ist betroffen — die Kalenderansicht (`/api/events`) zeigt weiter beide Farb-Chips. Default-Verhalten: Bei Vorrang 0 überall werden Dubletten trotzdem kollabiert (Gewinner per `source_id`); Roland setzt die Xalt-Quelle auf höheren Vorrang als MoreValue, um Xalt gezielt gewinnen zu lassen. Tests: `tests/test_feed_dedup.py` (reine Logik) + `tests/test_feed.py::TestFeedDeduplication` (Feed-Integration), Admin-API in `tests/test_admin_api.py`, E2E in `tests/e2e/test_admin.py::TestFeedPriority`.

## Belegt-Sync: MoreValue → Xalt (Etappe 22, erster SCHREIBZUGRIFF)

- **Zweck:** Rolands Xalt-Kollegen sollen in Xalts Frei/Belegt sehen, wann er wegen eines MoreValue-Termins nicht verfügbar ist. Das Add-on schreibt je MoreValue-Termin einen neutralen Block **„Busy MV"** in Rolands **primären** Xalt-Google-Kalender (`primary`). Fenster **[heute, heute+180 Tage)**, **alle** Termine der gewählten Quelle(n) inkl. ganztägige, Vergangenes nie. Fester Titel „Busy MV", `transparency:opaque` (= belegt), `visibility:private`, **keine** Beschreibung/Location (kein Termininhalt leakt). Einweg, kein Zurückschreiben in MoreValue.
- **Token-Trennung (Sicherheit):** Neuer Scope `WRITE_SCOPE = calendar.events` (`app/google_oauth.py`, `build_auth_url(scope=...)` wiederverwendet). Der Schreib-Token liegt in einer **separaten** Datei `google_busywrite_token.json` (0600, atomar wie `save_tokens`) — die readonly-Xalt-Lese-Token bleiben unangetastet. Refresh/401-Retry aus `app/sources/google.py` wiederverwendet. Admin-Flow Copy-Paste wie bestehend; nach Token-Tausch **keine** Kalenderliste (Ziel fest `primary`). Ein Workspace-Admin kann den Scope blockieren → deutsche Fehlermeldung, kein Crash.
- **Invariante (absolut, getestet): Das Add-on fasst AUSSCHLIESSLICH eigene, markierte Events an.** Jeder Block trägt `extendedProperties.private.familienkalender_busy = "<source-key>"` (`MARKER_KEY` in `app/google_busy.py`). Update/Delete gehen NUR über eine Google-eventId aus der Mapping-Tabelle (also stets ein selbst erzeugter Block). Reconciliation orphaner Blöcke listet **nur die eigenen** Blöcke per `events.list?privateExtendedProperty=familienkalender_busy=...` — NIE ein Vollscan; ein fremdes Event wird nie gelesen, geschweige denn geschrieben/gelöscht. `app/google_busy.py::BusyWriteClient` (insert/patch/delete/list_own_blocks), `event_body()` baut den neutralen Body.
- **Mapping-Tabelle** `busy_blocks (source_key PRIMARY KEY, google_event_id, start, end, all_day, updated_at)` (`app/storage.py`, Modell `BusyBlock`). `source_key = source_id|uid|<encoded-start>` = dieselbe Identität wie die events-Tabelle (`busy_sync.source_key`, nutzt `storage._encode_moment`).
- **Diff-Logik** (`app/busy_sync.py::run_busy_sync` → `_reconcile`): (1) desired = Events der gewählten Quellen im 180-Tage-Fenster (Start ≥ heute); je Key insert (neu) bzw. patch (Zeiten geändert), Mapping upsert. (2) gemappte Blöcke, deren Quell-Event weg/aus dem Fenster ist → delete + Mapping-Zeile weg. (3) Orphan-Reconciliation: eigene Blöcke ohne gültige Mapping-Zuordnung → delete (delete ist idempotent; 404/410 = ok). Frisch eingefügte Blöcke werden über `known_ids` NICHT als Orphan missverstanden. Läuft im periodischen Sync **nach** dem Einlesen der Quellen (`app/sync.py`), NUR wenn aktiviert UND Schreib-Token vorhanden; Fehler sind isoliert (sanitisiert in `busy_sync_status`, blockieren den Kalender-Sync nie).
- **Dubletten-Ausschluss (getestet) — BEIDE Read-Clients überspringen „Busy MV":** Der Familienkalender liest Rolands Xalt readonly und würde die eigenen „Busy MV"-Blöcke sonst zurücklesen → Dublette zu den MoreValue-Events. LÖSUNG im Google-Read-Client `app/sources/google.py`: Events mit `extendedProperties.private.familienkalender_busy` (primär) ODER Titel „Busy MV" (defensiver Fallback) werden übersprungen (`_is_own_busy_block`), landen nie in der DB → nie in Ansicht/Feed/Abo. **Etappe 24 (Loop-Fix):** Roland betreibt ZUSÄTZLICH ein externes Tool, das seinen Xalt-Kalender zurück nach MoreValue (Nextcloud) synct — dadurch kommen die „Busy MV"-Blöcke über CalDAV zurück und würden ungefiltert (a) als Dublette erscheinen und (b) vom Belegt-Sync erneut nach Xalt gespiegelt (Mirror-Loop). Deshalb überspringt jetzt AUCH der CalDAV-Read-Client (`app/sources/caldav.py`, in `_extract_events`) Termine mit Titel „Busy MV". CalDAV/iCal trägt keinen `extendedProperties`-Marker (und das externe Tool kopierte ihn ohnehin nicht mit), daher rein titelbasiert. Der Titel ist zentral in `app/models.py` (`BUSY_BLOCK_TITLE = "Busy MV"` + `is_busy_block_title()`: getrimmter, case-insensitiver EXAKTER Vergleich — „Busy MV Vorbereitung" bleibt erhalten) und wird vom Write-Body (`app/google_busy.py::BUSY_TITLE`) sowie beiden Read-Clients genutzt (keine Magic-String-Duplikation). **Bekannte Annahme:** greift nur, solange das externe Tool den Titel „Busy MV" UNVERÄNDERT übernimmt; ein Umbenennen/Präfix würde den Titel-Skip aushebeln. Tests: `tests/test_caldav.py::TestBusyBlockSkip` (Skip, Normalisierung, Enthält-Kein-Treffer), Read-Skip Google in `tests/test_google.py`.
- **Settings** (`app/settings.py`): `busy_sync_enabled` (Default aus), `busy_sync_source_ids` (JSON-Liste), `busy_sync_status` (JSON `{last_run, active_blocks, error}`, error sanitisiert).
- **Admin-UI** (`app/static/admin/busy-sync.js`, Sektion „Belegt-Sync (MoreValue → Xalt)", Etappe 23 überarbeitet): Reihenfolge (a) Schreib-Verbindung herstellen/trennen (Assistent mit `calendar.events`-Scope) + Verbindungsstatus, (b) Quellen-Auswahl klar gelabelt „Welche Kalender nach Xalt spiegeln:" mit Checkboxen der vorhandenen Quellen (erwartet „Roland MV") + eigenem „Auswahl speichern"-Button, (c) **prominenter Ein/Aus-Button** `#btn-busy-toggle` (kein Checkbox-Look mehr — verwechselte sich zuvor mit den Quellen-Checkboxen) mit Zustandstext „Belegt-Sync ist AUS – einschalten" bzw. „…ist AN – ausschalten", grün hervorgehoben wenn AN (`--ok-bg`/`--ok-text`, `aria-pressed`), sendet bei Klick sofort `PUT /api/admin/busy-sync` mit geflipptem `enabled` (aktuelle Quellenauswahl bleibt erhalten), (d) Status (letzter Lauf, aktive Blöcke, Fehler). ## Quellenarten & Geburtstage aus Google-Kontakten (Etappe 15)

- **Quellenarten** (`SOURCE_TYPES` in `app/models.py`, `type`-Spalte in `sources`): `google` (Google Calendar), `caldav` (Nextcloud/generisch), `google_contacts` (Kontakt-Geburtstage via Google People API). Alle teilen sich dieselbe Quellen-Infrastruktur (name, config, enabled, display_mode, color, shortcode, include_in_feed) und durchlaufen denselben Upsert/Filter/Feed-Pfad.
- **Warum People API:** Der Google-„Geburtstage"-Kalender ist über die Calendar API **nicht** abrufbar (Google-Limitierung). Kontakt-Geburtstage kommen daher aus der People API (`people.connections.list`, `personFields=names,birthdays`, `pageSize=1000`, Pagination). Dafür braucht der OAuth-Flow den Scope `https://www.googleapis.com/auth/contacts.readonly` (kein Calendar-Scope nötig). People API muss im Google-Cloud-Projekt aktiviert sein.
- **OAuth-Scope parametrisierbar** (`app/google_oauth.py`): `build_auth_url(client_id, scope=...)`; Default `CALENDAR_SCOPE` (Kalender-Flow unverändert), Geburtstage-Flow nutzt `CONTACTS_SCOPE`. Copy-Paste-Flow, Token-Storage (`google_token_<source_id>.json`, 0600) und Refresh identisch zum Kalender. Beim contacts-Connect wird **keine** Kalenderliste abgefragt (People API hat keine Kalender) — die Quelle wird direkt aus den geparkten Tokens angelegt.
- **People-Client** (`app/sources/google_contacts.py`): Token-Refresh + 401-Retry und Schutzlimits (`app/sources/limits.py`, `MAX_PAGES`) werden aus `app/sources/google.py` wiederverwendet. Aus jeder Person mit Namen und strukturiertem `birthdays[].date` (Monat/Tag, Jahr optional) entsteht je Jahr im Sync-Fenster (−7/+90 Tage) **ein ganztägiger** `CalendarEvent`. Personen ohne Namen oder ohne Geburtstag werden übersprungen; kaputte Einträge isoliert (Fehler gezählt, geloggt ohne Klartext). Da all_day=True, gelten sie **immer** als familienrelevant und erscheinen auch in gefilterten Ansichten und (optional) im Feed.
- **Design-Entscheidungen:** Titel `🎂 <Name>` (Torte-Emoji + Name); **kein Alter**, auch wenn das Geburtsjahr bekannt ist (viele Kontakte haben kein Jahr → inkonsistent; Alter auf gemeinsamem Kiosk unerwünscht). Geburtstage **ohne Jahr** (nur Monat/Tag) werden normal behandelt. **29.2.** fällt in Nicht-Schaltjahren auf den **28.2.** (letzter gültiger Tag), damit er nie verschwindet. Stabile UID `"<resourceName>|<jahr>"` → Feed/Storage sehen Updates als Updates.
- **Sync-Integration** (`app/sync.py` `_fetch_source_events`): bei `source.type == "google_contacts"` wird der People-Client genutzt (Token-Datei wie Google-Kalender), Fehlerisolation je Quelle wie bestehend.
- **Fest `full` (kein Anzeige-Modus):** `google_contacts`-Quellen sind immer `display_mode="full"`. Da Geburtstage ganztägige Events sind und ganztägige Events in `is_family_relevant` (app/filtering.py) IMMER als relevant gelten, hätte der Anzeige-Modus keinerlei Wirkung. Deshalb: kein Anzeige-Selector in der Quellenzeile (`sources.js`) und im Wizard; `create_source` erzwingt `full` (Request-Wert ignoriert); ein `display_mode`-PATCH auf eine `google_contacts`-Quelle wird stillschweigend ignoriert (alle anderen Felder bleiben änderbar).
- **Admin-UI:** Button „Geburtstage (Google-Kontakte) hinzufügen" (`app/static/admin/birthdays-wizard.js`, Endpunkte `POST /api/admin/google/contacts-auth-url` + `.../contacts-connect`). Assistent: App-Zugangsdaten (falls nötig) → Anmeldung mit contacts-Scope → Code einlösen → Quelle anlegen (Name-Vorschlag „Geburtstage", kein Anzeige-Modus da fest `full`, `include_in_feed` Default aus für `full`). `delete_source` räumt die Token-Datei auch für `google_contacts` auf.
