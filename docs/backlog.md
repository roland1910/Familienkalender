- **Risiko-Akzeptanz `homeassistant_api: true` (Nacharbeit Etappe 7):** Das Add-on läuft mit vollem HA-API-Zugriff (Read+Write) statt einer feineren Berechtigung — Home Assistant kennt für Add-ons keine granularere Abstufung als „ganz oder gar nicht"; ein rein lesendes API-Scoping existiert nicht. **Angreifermodell:** Wer Code-Execution im Add-on-Container erreicht (z. B. über eine Schwachstelle in einer Python-Dependency), hat damit automatisch vollen Zugriff auf die HA Core API des Haushalts — inklusive Schreibzugriff auf beliebige Entitäten, Automationen und Konfiguration, nicht nur Lesezugriff auf die paar Strom-Sensoren, die die App tatsächlich braucht. Der SUPERVISOR_TOKEN ist danach das eigentliche Kronjuwel des Containers. **Mitigationen (bereits umgesetzt, siehe oben in diesem Dokument):** (1) Die App selbst nutzt die API ausschließlich lesend (`GET states/{entity_id}`) — es gibt im Code keinen Schreibpfad, der Ausnutzung würde also eine zusätzliche Schwachstelle brauchen, die App-Logik missbraucht statt sie nur zu lesen; (2) Erreichbarkeit nur über HA-Ingress plus IP-Allowlist-Middleware (`ClientIPAllowlistMiddleware`), kein direkter Netzwerkzugriff auf den Container von außen; (3) AppArmor-Profil schränkt Dateisystem/Capabilities des Containers ein (mount/ptrace verboten, App-Code read-only); (4) gepinnte, gehashte Dependencies (`pip install --require-hashes`) reduzieren die Angriffsfläche für Supply-Chain-Kompromittierung, die der eigentliche realistische Weg zu Code-Execution wäre. **Bewusst nicht umgesetzt:** ein Long-lived Access Token mit eingeschränkten Rechten anstelle des SUPERVISOR_TOKEN wäre kein Sicherheitsgewinn — HA erlaubt auch für long-lived Tokens keine feinere Scope-Einschränkung als „voller API-Zugriff des Nutzers, der ihn erstellt hat"; der Rest bliebe identisch (Code-Exec = volle Kontrolle), nur mit einem zusätzlichen manuell zu pflegenden Secret statt des vom Supervisor verwalteten Tokens. Akzeptiertes Restrisiko, getragen durch die genannten Mitigationen.
